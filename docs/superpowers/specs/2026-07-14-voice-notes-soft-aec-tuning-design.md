# 软件回声消除调优：离线清洗 + 实时预对齐（两期）— 设计

日期：2026-07-14
状态：已确认（一期离线清洗先行，二期把延迟估计器移植到实时链路）

## 背景与根因

「保持外放音量」开启时（用户常开），录音不走 VPIO（macOS 强制绑定 ducking，
最小档也无法归零），改用普通 cpal 麦克风 + 软件回声消除（WebRTC AEC3，
`audio/aec.rs`）。该链路两个已实锤的短板：

1. **回音**：蓝牙外放延迟 300~600ms+，远超 AEC3 延迟估计范围（约 250ms），
   软件消回声完全失效，对方声音近乎全量串进 mic 轨（2026-07-08 蓝牙实录
   互相关实锤 lag≈600ms；`audio/mod.rs` 注释）。内置扬声器场景在 v0.3.2
   重采样相位修复（4b84e6b）后 AEC3 已基本收敛（文本级回声去重触发次数
   从每场 470~620 次降到 21 次）。
2. **噪音**：普通麦克风路径降噪模块（NS）未开；AGC2 自适应增益把底噪抬到
   约 -44dBFS（`aec.rs` 注释，设计取舍）。

回声是声学路径（扬声器→空气→麦克风）产生的，在「不降外放音量」约束下，
应用内唯一可行方向就是软件消回声——本设计不换路线，做两件事：把延迟
对齐搬到 AEC3 擅长的区间；把 NS 打开。

## 用户决策（约束）

- **自动检测、运行时实测延迟，不做静态配置**：不用 `experimental-aec3-config`
  硬扩延迟窗，不加用户设置项。
- **两期都做**：一期离线清洗治录音回放；二期移植估计器到实时链路治转写。
- **落盘单版本**：清洗后进转码，m4a 即干净版；原始 WAV 照原流程删除。
- 录音回放不要求停录后立即可用——离线清洗有时间预算。

## 架构总览

```
一期（离线）
停录 → 转码 worker 出队（已持笔记锁）
  → 清洗工序：mic.wav + system.wav 都在 且 本场走软件 AEC 路径
      ① 读两轨 PCM（16k 单声道 s16le）
      ② 延迟估计（§延迟估计器）→ 置信度不足 → 跳过清洗
      ③ system 参考按估计值前移对齐
      ④ APM 离线重跑：AEC3 + NS(High)，AGC 不开；暖机一遍丢输出，正式一遍取输出
      ⑤ mic.cleaned.tmp → fsync → 原子 rename 覆盖 mic.wav
  → 原有转码照旧（压 m4a、时长校验、删 WAV）

二期（实时）
system 采集 → [新] 预延迟环形缓冲 → AecRender.push（参考）
mic 采集 → AecCapture.process（消回声，链路加开 NS）
后台估计 worker：每 5s 用最近 20s 包络重估延迟，驱动预延迟调整
```

### 一期：离线清洗工序

- 位置：`store/transcode.rs` worker 内、encode 之前。worker 已持笔记锁，
  清洗-替换-转码全窗口互斥，无新增并发面。
- 触发条件（全自动，零配置）：
  - `mic.wav` 与 `system.wav` 同时存在（`record_system_only` 场景无 mic 轨，
    VPIO 场景无需清洗——按会话录制时所走路径标记判定，标记随轨道 meta 落盘）；
  - 延迟估计置信度过门限。内置扬声器场景残余回声弱、互相关峰不显著，
    自然跳过，不动已经干净的音频。
- 处理参数：AEC3 默认窗（对齐后延迟已在窗内）+ NS High；AGC2 不开
  （录制时已做过增益，清洗不再动电平）。
- 暖机：AEC3 有开头收敛期，离线负担得起双 pass——第一遍喂完丢输出，
  第二遍取输出。
- 落盘：写 `mic.cleaned.tmp`，fsync 后原子 rename 覆盖 `mic.wav`，
  再进原有转码。任何一步失败：打日志、跳过清洗、原样转码。
- 可观测：清洗结果（是否执行、延迟估计值、置信度、分段数）写入轨道 meta，
  并 eprintln 进 stderr.log。

### 延迟估计器（两期共用核心）

新模块 `audio/delay_estimate.rs`，纯函数、无状态：

- 输入：两条 16k 单声道 PCM 切片（参考 = system，观测 = mic）；
- 输出：`(delay_ms, confidence)`（落地时增补第三字段 `peak`——主峰 NCC 绝对值：
  无关信号的比值噪声大不可单用，判别改为 confidence+peak 双门限，标定值见
  实现注释与计划偏差记录）；
- 实现：10ms 能量包络 → 归一化互相关，搜索 0~1200ms；主峰位置为延迟，
  主峰/次峰比为置信度；置信度门限不拍脑袋——实施期用真实录音标定
  （蓝牙场次应过门限、内置扬声器场次应被拒），定值写进模块注释；
- 漂移处理（离线）：按 60s 窗重估。相邻窗差 <40ms → 全程单值；
  超过 → 分段应用。~~蓝牙时钟漂移通常很小，预期绝大多数会议走单值路径~~
  **（假设被 2026-07-14 标定实测推翻：正例C（2026-07-08 蓝牙实锤场，具体
  笔记见本地标定报告，不入库）延迟 250~1200ms 散布，3 窗撞 1200ms 搜索
  上限，真实延迟或越出搜索域。稳定延迟场为主效场景；漂移场一期仅部分
  清洗——过闸窗定段、漏清不伤近端，二期实时滑窗重估为根治路径。）**

### 二期：实时预对齐

- 预延迟环形缓冲插在 system 采集与 `AecRender.push` 之间，初值由现有
  `default_output_is_bluetooth()` 探测给：蓝牙 ≈450ms，其他 0。
- 后台估计 worker：每 5s 对最近 20s 包络跑同一估计器；panic 隔离
  （catch_unwind，与 HookBus 同哲学）。
- 调整策略：仅当置信度过门限 且 与当前预延迟差 >80ms 时调整；调整点
  对参考流做丢弃/补零，AEC3 随后重收敛（几百 ms，期间残余回声由现有
  文本去重兜底）。低置信度永不动时间轴——行为等同现状。
- 实时链路同时开 NS；`stats().delay_ms` 每 10s 打日志作观测面。

## 错误处理

与转码 worker 同一哲学：音频增强是转写之上的增值层。

- 离线：清洗任一步失败（读轨、估计、APM、写盘）→ 日志 + 跳过清洗 +
  原样转码。绝不因清洗丢录音、不阻塞停录。
- 实时：估计 worker 挂掉/低置信度 → 预延迟维持现值；缓冲初值探测失败
  按非蓝牙（0ms）处理，等同现状。

## 测试与验收

单测（复用 `aec.rs` 的确定性噪声/功率工具）：

- 估计器：合成 600ms 延迟回声 → 误差 <20ms；纯无关噪声 → 低置信度；
  分窗漂移（前半 600ms 后半 660ms）→ 正确分段。
- 离线清洗端到端：近端 = 远端 600ms 延迟拷贝 + 本地语音 → 清洗后回声
  能量降 ≥15dB，本地语音功率保持（±3dB；落地放宽为 ±9dB——NS(High) 对无谐波
  结构的合成噪声过度抑制，系合成信号局限，真实语音行为由真实录音标定兜底，
  见计划偏差记录）。
- 触发条件矩阵：缺 system 轨 / VPIO 会话 / 低置信度 → 不动 mic.wav
  （字节级一致）。
- 二期：预延迟缓冲样本守恒；调整点行为；估计 worker panic 不影响录制。

真实数据验收：

- 2026-07-08 蓝牙实录（正例C,m4a 解码回 PCM）跑离线清洗：对照试听 +
  清洗前后 mic 轨残余互相关包络峰对比（目标 ≥15dB 改善）。
- 内置扬声器的近期录音跑清洗：应被置信度门限跳过。

回归：现有全部测试（412+）保持绿。

## 风险与非目标

- AEC3 离线使用属非典型用法（引擎为实时设计），靠暖机双 pass 弥补；
  若清洗后仍有可闻残余，将来可选深度学习增强（DeepFilterNet 类）做三期
  polish——本期非目标。
- 蓝牙麦克风（HFP 16kHz）音质天花板是采集侧物理限制，本设计不解决
  设备选择问题（不自动切换输入设备）——非目标。
- 不改 VPIO / ducking 路线；不加任何用户配置项。
