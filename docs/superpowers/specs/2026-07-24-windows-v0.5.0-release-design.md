# Windows v0.5.0 Release Design

## Goal

Publish the verified Windows build as official downloadable assets on the existing GitHub `v0.5.0` release, and update the README so Windows users have accurate installation and security guidance.

## Release Identity

- Keep the application version at `0.5.0`.
- Keep the existing `v0.5.0` Git tag unchanged.
- Do not rewrite or force-move the published tag.
- Build the Windows artifacts from the latest commit on PR #57.
- Add an explicit note to the existing release description that the Windows artifacts were built from PR #57 after the original tag.

This preserves the integrity of the published tag while honoring the requested `v0.5.0` application version.

## README Changes

Update the Chinese README installation sections to:

- State that official Windows 10/11 x64 installers are available from GitHub Releases.
- List `voice-notes_0.5.0_x64-setup.exe` as the recommended installer.
- List `voice-notes_0.5.0_x64_en-US.msi` as the managed-deployment alternative.
- Remove statements that Windows users must build from source.
- Explain that the current Windows binaries are unsigned and may trigger Windows Security, Smart App Control, or SmartScreen.
- Recommend downloading only from the official `SoulZhong/voice-notes` release page and checking the published SHA-256 checksums.
- Avoid instructing users to disable Windows security features globally.
- Keep source-build instructions for developers.

Because the current README contains visibly corrupted Chinese text in several sections, the release change will make focused edits to the Windows installation and FAQ areas instead of rewriting unrelated documentation.

## Windows Artifacts

Build with the repository's Windows Tauri configuration and the already verified Windows dependency environment.

Publish:

- `voice-notes_0.5.0_x64-setup.exe`
- `voice-notes_0.5.0_x64_en-US.msi`
- `SHA256SUMS-windows.txt`

The checksum file will contain SHA-256 hashes for both installers and will be generated from the exact files uploaded to GitHub.

## Verification

Before uploading:

- Run the full frontend test suite.
- Run Svelte/TypeScript checks.
- Run `cargo check` with the Windows build environment.
- Build the release executable, NSIS installer, and MSI.
- Confirm the installers exist, are non-empty, and report their sizes and SHA-256 hashes.
- Silently install the NSIS package.
- Launch the installed application and confirm the process is alive and responsive.
- Repeat the AI-to-other-tab navigation smoke test.

After uploading:

- Fetch the `v0.5.0` release metadata.
- Confirm all three Windows assets are present with non-zero sizes.
- Confirm the release remains tagged `v0.5.0`.
- Confirm the macOS assets remain present.

## Git and PR Flow

- Commit the README update, design, implementation plan, and any release-support files to `codex/windows-icon-unification`.
- Push the branch so PR #57 updates.
- Do not merge PR #57 as part of this request.
- Build and upload from the latest pushed PR commit.
- Update the existing `v0.5.0` release notes to identify that commit as the source of the Windows artifacts.

## Failure Handling

- If tests or compilation fail, do not upload artifacts.
- If packaging succeeds but installation or smoke testing fails, do not upload artifacts.
- If an asset with the same filename already exists, inspect it before replacement; replace only the intended Windows asset.
- Never move or recreate the `v0.5.0` tag.
- Never delete existing macOS release assets.
