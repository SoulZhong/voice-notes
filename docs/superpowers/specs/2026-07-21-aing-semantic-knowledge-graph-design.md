# Aing 语义知识图谱一期：关系、治理与探索网络设计

日期：2026-07-21  
状态：设计已确认  
上游：`2026-07-16-aing-pipeline-design.md`、`2026-07-17-aing-p2b-graph-page-design.md`

## 背景

现有 Aing 图谱已具备：

- 每篇 `aing.json` 的局部实体与正文提及区间。
- SQLite `entities` / `note_entities` 派生索引。
- 实体共现图、文章共享实体图、相关笔记、实体详情与实体改名。
- 力导图缩放、逐层展开、搜索、类型过滤和大图降级。

当前边只表达“共同出现在一篇笔记”，不能回答“为什么有关”“谁负责什么”“某项决定由哪场会议产生”。实体身份仍主要按规范名解析，无法完整处理同音人名、同名异物、错误合并与证据级拆分。图谱已经有探索外形，但缺少可核验的语义和稳定的人工治理层。

一期把三个部分作为一个闭环交付：

1. **语义关系**：Aing 抽取有方向、有类型、有证据的关系。
2. **实体治理**：用户能改名、维护别名、合并、证据级拆分、关联会议搭子，并纠正关系。
3. **探索网络**：语义关系成为主图，共现降为弱底图，支持逐层探索、证据查看和两点寻路。

## 目标

- HTTP 与本机 Agent 两种 Aing 方式都能产出同一结构的实体、关系和证据。
- 每条 AI 关系都能回到具体笔记、段落和原文区间核验。
- 高置信关系直接进入主图；低置信、冲突或自定义关系进入“待整理”。
- 人工裁决在重新 Aing、关系补建和 SQLite 整库重建后保持不变。
- 非人物实体使用稳定身份；改名不改 ID，同名同类型实体可以并存。
- 旧笔记可批量补建关系，无需重新修订全文。
- 主图默认表达当前有效关系，并可查看历史轨迹。
- 用户可沿关系逐层展开，并查看两个实体之间的最短可解释路径。
- 无模型、抽取失败、关系为空或索引损坏都不影响录音、转写、修订稿和现有共现图。

## 非目标

- 不做录音中的实时增量 Aing；仍属于 Aing Phase 3。
- 不做多用户、云同步、远端图数据库或权限系统。
- 不做自由本体编辑器；一期使用稳定核心关系类型与自定义关系兜底。
- 不做自动云端补建；任何历史笔记调用都由用户明确发起。
- 不做无证据的 AI 推理边；模型无法定位证据就不能把关系发布到图谱。
- 不做时间轴动画、图查询语言或多路径排名工作台；一期只做时间过滤、当前/历史切换和最短可解释路径。
- 不让 Agent 直接写人工裁决；Agent 产物始终属于模型事实。
- 不以 SQLite 为最终真值；它仍是可替换、可整库重建的派生索引。

## 已拍板的关键决策

1. 一期按“闭环可用”验收，三个部分一起设计、同一期交付。
2. HTTP 与 Agent 同期能力对等。
3. 模型事实保存在每篇 `aing.json`；人工裁决保存在全局 `knowledge-overrides.json`。
4. 关系使用“核心类型表 + 自定义关系”，全部有方向。
5. 每条 AI 关系强制携带精确证据、模型信息和置信度。
6. 历史笔记支持关系专用批量补建，不重新润色全文。
7. 高置信且无冲突的关系直接发布；其余进入待整理或仅保留在模型事实中。
8. 实体支持证据级拆分，关系随证据重新归属。
9. 主图表达当前状态，关系详情保留历史轨迹。
10. 视觉采用“语义主图 + 共现弱底图”；实体详情内治理，疑点进入轻量待整理入口。
11. 一期支持逐层展开、关系/时间过滤、搜索定位和两点寻路。

## 架构总览

```text
HTTP 结构化响应 ─┐
                  ├─> 共享关系契约与校验器 ─> 每篇 aing.json 模型事实
Agent MCP 写回  ──┘                                  │
                                                     ▼
                                        稳定实体与关系解析器
                                                     ▲
                                                     │
                                    knowledge-overrides.json
                                     稳定身份 + 人工裁决账本
                                                     │
                                                     ▼
                                   graph.sqlite 临时重建并原子替换
                                                     │
                          ┌──────────────────────────┼──────────────────────┐
                          ▼                          ▼                      ▼
                    语义关系主图               待整理/治理             相关笔记/寻路
```

解析顺序固定为：

```text
所有 aing.json 模型事实
→ 稳定实体解析
→ knowledge-overrides.json 人工裁决
→ 当前关系与历史关系计算
→ graph.sqlite
```

模型重跑只能替换模型事实，不能修改、删除或绕过人工裁决。

## 组件边界

### 1. 关系契约与校验器

负责 HTTP/Agent 共用的输入结构、关系类型规范、证据验证、重复合并、置信度分层与错误报告。它不读写 SQLite，不做全局实体解析。

### 2. Aing 关系抽取器

- HTTP 路径：在现有“修订 + 实体”结构化响应中增加关系。
- Agent 路径：通过受约束 MCP 读工具取得 Aing 上下文，再通过受约束写工具提交实体、关系和证据。
- 补建路径：只读取现有修订段落与实体，运行关系专用提示词，不重新修改正文。

### 3. 知识裁决存储

`knowledge-overrides.json` 保存稳定非人物身份、兼容映射、提及绑定、合并重定向、关系裁决和操作历史。它是全局持久真值，写入必须持跨进程锁并原子替换。

### 4. 稳定解析器

把每篇局部实体和关系解析到全局稳定 ID，重放人工操作，计算当前/历史状态，并输出与存储无关的规范图谱。

### 5. 索引构建器

从规范图谱生成新的 SQLite 临时库，完整成功后替换现有索引。构建失败继续使用上一份可读索引。

### 6. 查询与交互层

提供语义图数据、关系详情、证据、待整理、治理命令、时间过滤和两点寻路。图谱查询失败只降级图谱功能，不向笔记主链传播。

## 每篇 `aing.json` 数据模型

现有文档 schema 增加稳定 mention ID、`stages.relations`、`graph_extraction` 和 `relations`。旧文档缺字段时全部按默认值读取；读取旧 mention 时由后端按同一哈希规则即时补出 ID，不要求先重写文件。

```json
{
  "schema_version": 2,
  "stages": {
    "filter": "done",
    "recluster": "done",
    "llm": "done",
    "entities": "done",
    "relations": "done"
  },
  "graph_extraction": {
    "contract_version": 1,
    "provider": "openai",
    "model": "model-name",
    "run_id": "run_hash",
    "generated_at": "RFC3339",
    "source_hash": "sha256",
    "mode": "full_aing"
  },
  "paragraphs": [
    {
      "source_seqs": [12, 13],
      "text": "张三负责灯塔计划",
      "mentions": [
        {"id":"mn_subject","entity":"ent_1","start":0,"end":2},
        {"id":"mn_object","entity":"ent_2","start":4,"end":8}
      ]
    }
  ],
  "relations": [
    {
      "id": "rf_hash",
      "subject": "ent_1",
      "predicate": {
        "type": "responsible_for",
        "label": null
      },
      "object": "ent_2",
      "subject_mentions": ["mn_subject"],
      "object_mentions": ["mn_object"],
      "confidence": 0.92,
      "valid_from": null,
      "valid_to": null,
      "evidence": [
        {
          "id": "ev_hash",
          "paragraph_index": 4,
          "start": 5,
          "end": 18,
          "quote": "张三负责灯塔计划",
          "source_seqs": [12, 13],
          "source_hash": "sha256"
        }
      ]
    }
  ]
}
```

### 关系事实 ID

- Mention ID：`mn_` + SHA-256 前 24 个十六进制字符，输入为 `note_id + source_seqs + local entity id + start + end + 对应正文`。
- 证据 ID：`ev_` + SHA-256 前 24 个十六进制字符，输入为 `note_id + source_seqs + start + end + 规范化 quote`。
- 模型关系 ID：`rf_` + SHA-256 前 24 个十六进制字符，输入为 `note_id + subject local id + predicate + object local id + 排序后的 subject/object mention ids + evidence ids`。
- 相同事实重新抽取时 ID 稳定；证据真实变化时生成新事实，并由解析器与人工裁决决定是否取代旧版本。

### 证据定位

`paragraph_index` 与字符区间用于正常情况下直接定位；`source_seqs`、原文快照和哈希用于重新分段后的恢复。字符区间使用 Unicode scalar value 下标，与现有 mention 口径保持一致。每条 AI 关系必须引用至少一个主体 mention 和一个客体 mention；这是证据级拆分后关系能准确跟随实体身份的依据。

## 核心关系类型

一期内置八个稳定类型；主体和客体种类不符合常见方向时进入待整理，但不直接丢弃。

| 类型代码 | 中文显示 | 典型方向 |
|---|---|---|
| `participates_in` | 参与 | 人/组织 → 项目/会议/事件 |
| `responsible_for` | 负责 | 人/组织 → 项目/任务 |
| `belongs_to` | 属于 | 人/项目/产品 → 组织/上级项目 |
| `uses` | 使用 | 人/项目/组织 → 技术/工具/产品 |
| `depends_on` | 依赖 | 项目/任务/产品 → 任意实体 |
| `produces` | 产生 | 会议/项目/组织 → 决议/任务/成果 |
| `assigned_to` | 指派给 | 任务/事项 → 人/组织 |
| `occurs_at` | 发生于 | 会议/事件/任务 → 地点/日期 |

无法规范到上述类型时使用：

```json
{"type":"custom","label":"推动"}
```

自定义关系必须有非空、去首尾空格后的显示名，并默认进入待整理。

## `knowledge-overrides.json` 数据模型

该文件同时保存稳定身份注册表和可重放人工操作：

```json
{
  "schema_version": 1,
  "registry": {
    "kg_hash": {
      "kind": "project",
      "name": "灯塔计划",
      "aliases": ["Lighthouse", "灯塔"],
      "status": "confirmed"
    }
  },
  "legacy_ids": {
    "e:灯塔计划": "kg_hash"
  },
  "operations": [
    {
      "id": "op_hash",
      "at": "RFC3339",
      "kind": "rename_entity",
      "payload": {
        "entity_id": "kg_hash",
        "name": "灯塔计划"
      }
    }
  ]
}
```

### 稳定实体 ID

- 人物继续使用现有 `person_id`。
- 非人物使用 `kg_` + SHA-256 前 24 个十六进制字符。
- 首次自动建档的哈希种子为 `kind + 规范化名称 + 首次解析时的 note_id/local_entity_id`，分配后写入 registry，后续不因改名或出现更早笔记而改变。
- 用户拆分创建的新实体以拆分操作 ID 为哈希种子，允许同名同类型实体并存。
- 现有 `e:<规范名>` 写入 `legacy_ids`，旧深链和历史重定向继续解析。

### 人工操作类型

- `rename_entity`：修改规范显示名，ID 不变。
- `add_alias` / `remove_alias`：维护解析别名。
- `merge_entity`：旧 ID 重定向到目标 ID。
- `bind_mention`：把具体 mention ID 绑定到指定实体，用于拆分与人物关联。
- `confirm_relation`：确认模型关系。
- `edit_relation`：替换类型、方向、时间或说明。
- `suppress_relation`：按稳定主体、谓词、客体永久抑制模型关系，直到恢复。
- `end_relation`：设置关系结束时间，转入历史。
- `restore_relation`：撤销抑制或结束裁决。
- `create_entity` / `create_relation`：创建用户事实。
- `undo`：引用既有操作并写入补偿语义；历史操作不删除、不改写。

每条操作保存执行时间、前值摘要和后值；单用户本地应用不引入用户身份字段。

## 实体解析规则

按以下顺序解析，任何低层规则不能覆盖高层人工裁决：

1. 具体 mention/evidence 的 `bind_mention`。
2. 人物 `person_id` 绑定与人物重定向。
3. `merge_entity` 重定向链。
4. 已确认实体的规范名和别名精确匹配，匹配时同时考虑 kind。
5. legacy `e:<名称>` 映射。
6. 为未命中实体分配新的稳定 `kg_*` ID。

同音、编辑距离或模型相似度只产生候选，不自动绑定人物或合并实体。候选进入待整理。

## 证据级拆分

拆分流程不改写历史 `aing.json`：

1. 用户从实体详情选择“拆分实体”。
2. UI 按笔记分组列出所有 mention 证据，显示原文、时间和关系数量。
3. 模型可提供预选分组，但用户必须确认每个移动项。
4. 后端创建稳定新实体，并为选中的 mention ID 写入 `bind_mention`。
5. 解析器按关系事实的 `subject_mentions` / `object_mentions` 重新归属关系。
6. 同一模型关系的证据被分到不同身份时，规范图谱生成多条关系，各自只携带所属证据。

拆分提交前展示将移动的笔记、提及和关系数量；提交后可以通过补偿操作撤销。

## 关系当前状态与历史

- `valid_to` 为空或晚于当前时间的关系属于当前候选；已经 `end_relation` 或 `valid_to` 到期的关系进入历史。
- 人工确认与人工创建的关系优先于模型事实。
- 模型不能仅凭“出现了更新关系”自动结束旧关系；正文明确表达“改由、停止、替代”时可以提出时间变更，但时间冲突进入待整理。
- 主图默认只显示当前发布关系；关系详情展示当前版本、历史版本和全部证据。
- 用户确认负责人变更时，单次治理操作同时结束旧关系并确认新关系，保证主图立即收敛。

## HTTP 与 Agent 对等

### HTTP

现有结构化响应增加 `relations`。关系证据区间基于模型返回的修订后文本，由共享校验器对最终段落逐字验证。

### Agent

新增两个受约束 MCP 工具：

- `get_aing_context(note_id)`：返回修订段落、`source_seqs`、局部实体、mentions、核心关系类型与契约版本。
- `apply_aing_graph(note_id, entities, relations, contract_version, model)`：只写该篇模型实体/关系事实，不能写全局人工裁决。

Agent 流程固定为：

1. 读取笔记并完成现有文本修订。
2. 写回修订文本。
3. 重新读取 Aing context，确保关系证据针对最终文本。
4. 提交结构化实体和关系。

Agent 写回失败只标记实体/关系阶段失败，已经成功的文本修订保留。

## 关系校验与发布规则

共享校验器执行：

- 主体和客体必须存在于本篇局部实体集合。
- `subject_mentions` / `object_mentions` 必须非空、存在于本篇 mentions，且分别指向主体/客体局部实体。
- 禁止主体与客体相同的无意义自环。
- 核心类型代码必须在白名单；custom 必须有显示名。
- `confidence` 必须为 `0.0..=1.0`。
- 每条 AI 关系至少有一条证据。
- `paragraph_index` 必须存在；`start < end` 且区间逐字等于 quote。
- `source_seqs` 必须是该段 `source_seqs` 的非空子集或完整集合。
- 完全相同的主体、谓词、客体关系合并证据。
- 相同证据不能被同一关系重复计数。

发布分层：

- `confidence >= 0.80`、证据完整、属于核心类型、身份无冲突、时间无冲突：进入主图。
- `0.50 <= confidence < 0.80`、自定义类型、身份候选冲突或时间冲突：进入待整理。
- `confidence < 0.50`：保留在单篇模型事实中，不进入 SQLite 图谱索引。
- 人工确认、编辑或创建的关系不再受模型置信度阈值影响。
- `suppress_relation` 默认按稳定主体、规范谓词、稳定客体抑制，模型更换证据也不能复活同一关系。

## 批量补建关系

图谱页和 AI 页提供同一补建入口：

- 默认选择 `relations` 缺失、失败或契约版本过旧的笔记。
- 开始前显示笔记数量、执行方式、模型和隐私提示；不伪造无法准确计算的费用金额。
- 只读取现有修订文本与实体，不重新润色全文。
- 抽取发现缺失实体时可增补局部实体，但不能自动合并全局身份。
- 每篇成功结果独立原子替换该篇模型关系；失败不清空上一份成功关系。
- 使用现有全局串行 Aing 闸，提供进度、当前笔记、取消和恢复。
- 恢复时按 `source_hash + contract_version` 跳过已经完成且未变化的笔记。
- 首次启用不自动补建，版本升级也只标记“可补建”。

## 图谱索引

现有 `entities`、`note_entities`、重定向和名称覆盖数据迁移到新构建器的输出。新增概念表：

- `relations`：规范主体、谓词、客体、当前/历史状态、来源与置信度。
- `relation_evidence`：关系到笔记证据的多对多记录。
- `entity_mentions`：稳定实体到具体 mention evidence 的绑定。
- `pending_review`：疑似重复、低置信关系、自定义类型、时间/身份冲突。
- `graph_meta`：契约、解析器和构建版本。

SQLite 不保存唯一的人工真值。重建使用全新临时库，校验成功后原子替换；旧库在新库可用前保持服务。

## 治理交互

### 实体详情

实体详情包含“概览、关系、证据”三个区域：

- 规范名、类型、别名、确认状态和统计。
- 当前关系与历史关系。
- 按笔记分组的全部原文提及。
- 改名/别名、合并、拆分、新建关系、关联会议搭子入口。

右键菜单只提供相同动作的快捷入口，不承载独有能力。

### 合并

选择目标前展示双方规范名、别名、笔记数、提及数、关系数和人物身份。提交写入重定向，所有提及与关系在解析时汇入目标；模型事实不改。

### 待整理

侧栏常驻轻量“待整理 N”入口，按以下类别分组：

1. 疑似重复实体或人物匹配。
2. 低置信关系。
3. 自定义关系类型。
4. 时间或身份冲突。

每项支持确认、编辑、否决和稍后处理。否决写入持久抑制，不是仅隐藏本次结果。

### 关系详情

点击语义边打开关系抽屉，展示方向、类型、当前状态、置信度、抽取方式、模型、历史和全部原文证据。用户可以修改类型、方向、起止时间、说明，或者确认、结束、否决和恢复。

人工关系必须满足以下之一：

- 关联至少一条笔记证据；或
- 明确标记 `user_assertion`，UI 持续显示“用户直接声明”，不能伪装成来源于笔记的事实。

## 探索网络

### 视觉层级

- 语义关系：实线、方向箭头、完整关系名称，是主信息。
- 共现关系：淡色虚线、显示共同笔记数，是可隐藏弱底图。
- 节点：圆点与完整名称同心，不使用省略号或独立标签卡片。
- 关系名称：不截断；大图通过语义缩放控制显示数量，放大、悬停、聚焦和路径态逐级揭示。
- 低置信或未发布关系不混进主图，仅在待整理中出现。

### 交互

- 点击节点：聚焦并展开一层直接语义关系。
- 继续点击：逐层扩展；支持收起到默认骨架。
- 点击语义边：打开关系证据抽屉。
- 点击共现边：展示共同笔记，并提供“补建关系”或“新建关系”。
- 搜索：定位并聚焦命中实体。
- 过滤：实体类型、关系类型、时间范围和“包含历史关系”。
- 数据规模控制：默认显示关系最紧密骨架，保留逐层展开、显示全部和收起。
- 动效只解释布局变化；大图冷启动定格；`prefers-reduced-motion` 直接呈现结果。

### 两点寻路

1. 在实体菜单中设置路径起点。
2. 选择终点。
3. 在当前时间/类型过滤条件下计算最短路径。
4. 默认只使用当前发布的语义关系，以 Dijkstra 求最低代价路径：人工确认/人工创建边代价 `1.0`；高置信模型边代价 `1.2 + (1.0 - confidence)`；主动开启的共现弱连接代价 `3.0`。总代价相同时依次按跳数、更高最低置信度、稳定 ID 字典序裁决，保证结果可复现。
5. 路径结果逐边显示方向、类型和证据。
6. 用户可以主动开启“包含共现弱连接”；弱连接始终以虚线标记。

路径搜索不把共现边默认当作语义事实，也不因路径更短而绕过人工抑制。

### 空态与旧数据

- 无语义关系但有共现数据：继续显示现有共现图，并提示“尚未补建语义关系”。
- 无实体：沿用配置 Aing 的引导空态。
- 关系查询失败：退回共现图或上一次可读索引。
- 列表视图提供与画布等价的键盘入口、关系摘要和治理入口。

## 当前关系与历史关系

- 主图默认只显示当前发布关系。
- 已结束关系只在开启“包含历史关系”或关系详情历史中显示。
- 时间过滤同时作用于主图、实体详情、待整理和寻路。
- 模型提出的负责人替换等时间变化，如果会改变既有当前关系，必须进入待整理。
- 用户确认变更时，一个治理动作同时结束旧关系、确认新关系，避免中间状态误导。

## 错误处理

- HTTP/Agent 抽取失败：不覆盖上一份成功关系，文本修订不回滚。
- 单条结构不合法：返回字段级错误；整份载荷未通过校验时不落盘。
- 置信度不足：属于发布分层，不当作系统错误。
- SQLite 重建失败：继续使用上一份可读库并暴露降级状态。
- `knowledge-overrides.json` 损坏：不能静默忽略。图谱进入只读降级状态，保留上一份 SQLite 展示，暂停治理和补建，并提示从备份恢复。
- 人工裁决证据失效：先按 `source_seqs + source_hash + quote` 重定位；仍失败则进入“证据已失效”待整理，不删除裁决。
- 批量补建取消：当前未提交笔记不落半份结果，已经成功笔记保持可恢复状态。
- 任意图谱失败：只记录并降级，不影响录音、ASR、Aing 修订稿、回放和会议搭子。

## 写入与并发

- `aing.json` 模型事实继续使用笔记级锁和临时文件原子替换。
- `knowledge-overrides.json` 使用独立跨进程文件锁；GUI、MCP 和未来 CLI 共享同一写入口。
- Agent 的 `apply_aing_graph` 只能写单篇模型事实，不获取人工裁决写权限。
- SQLite 构建期间读取继续命中旧库；替换后查询句柄重新打开。
- 图谱命令不得持知识裁决锁执行模型调用、SQLite 重建或长时间 UI 查询。

## 测试策略

### Rust 纯测试

- schema v1/v2 兼容、默认字段和往返序列化。
- 核心类型、自定义类型、置信度范围和自环拒绝。
- Unicode 字符区间、quote 匹配、`source_seqs`、哈希与证据重定位。
- 相同关系证据合并、关系 ID 稳定和变化事实换 ID。
- HTTP/Agent 使用同一 fixtures 得到一致规范载荷。

### 解析器矩阵

- 别名命中、同名同 kind、同名不同 kind、人物绑定和 legacy id。
- 合并重定向链、证据级拆分、拆分撤销和关系随证据分裂。
- 确认、编辑、抑制、恢复、结束和用户声明关系。
- 模型重跑、契约升级和补建后人工裁决不变。
- 当前/历史状态和时间冲突进入待整理。
- 任意输入文件遍历顺序产生相同规范图谱。

### 存储与重建

- 跨进程写锁、有界竞争、临时文件和损坏恢复。
- SQLite 临时重建失败不影响旧库。
- 连续重建幂等，规范实体、关系、证据和待整理数量一致。
- 旧 `e:<名称>` 深链解析到稳定 ID。
- 批量补建取消、恢复、跳过和失败不清旧结果。

### 前端测试

- 关系/实体/时间过滤。
- 当前与历史切换。
- 逐层展开、收起和默认骨架。
- 最短路径、方向展示、共现弱连接可选加入。
- 待整理分类、关系证据抽屉和治理动作状态刷新。
- 证据级拆分的选择、预览、提交和撤销。

### 浏览器与真机验收

- HTTP 与 Agent 各完成一篇笔记，均产生可核验证据关系。
- 批量补建旧笔记不改变修订正文。
- 改名、别名、合并、拆分、否决和关系历史在重新 Aing 后保持。
- 实体详情、待整理、关系抽屉、键盘路径、亮暗主题和减少动效可用。
- 使用至少 1000 实体、5000 语义关系的固定 fixture 验证默认骨架、连续展开、寻路和显示全部不会卡死。
- 无模型、关系失败和 SQLite 损坏时，旧笔记与共现图仍可用。

## 一期验收标准

1. HTTP 与 Agent 对同一 fixture 生成同一关系契约并通过相同校验。
2. 主图不存在无证据 AI 关系。
3. 高置信关系、待整理和低于阈值事实严格分层。
4. 用户裁决在重新 Aing、补建和整库重建后逐项一致。
5. 同名同类型实体能通过证据级拆分稳定共存。
6. 实体改名不改变稳定 ID，旧深链仍可访问。
7. 主图可以从当前关系进入证据、历史、治理和两点路径。
8. 旧数据无需模型调用即可继续显示共现图。
9. 所有失败只降级图谱，不影响笔记核心链路。
10. 大图 fixture 通过交互与稳定性验收。

## 实施拆分约束

这是一份统一一期设计，但 implementation plan 应按可独立验证的依赖链拆任务：

1. 数据契约、证据校验和 `aing.json` 兼容。
2. 知识裁决账本、稳定实体 ID 与解析器。
3. 语义关系 SQLite 索引、临时重建和查询。
4. HTTP 关系抽取与关系专用补建。
5. Agent MCP 对等读写。
6. 实体详情治理、证据级拆分与待整理。
7. 语义主图、证据抽屉、过滤和逐层探索。
8. 两点寻路、大图性能、迁移与端到端验收。

各任务必须保留“模型事实、人工裁决、派生索引”三层边界；不能为了少写一层解析代码把人工修正塞回 `aing.json`，也不能把 SQLite 提升为唯一真值。
