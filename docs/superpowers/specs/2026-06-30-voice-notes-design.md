# 实时会议转写应用 — 设计文档

- **日期**：2026-06-30
- **状态**：已通过头脑风暴评审，待编写实现计划
- **代号**：voice-notes

## 1. 定位

一个 **macOS 优先、架构跨平台** 的实时会议转写桌面应用。同时采集**麦克风 + 系统声音**，本地用 **Whisper** 做中英混合识别，本地做**说话人区分（diarization）**，实时显示转写文本并保存为可检索、可导出的会议笔记。

**核心原则：全程离线、音频不上传；会议进行中绝不丢内容。**

### 已确认的需求约束

| 维度 | 决策 |
|---|---|
| 主场景 | 会议 / 长音频转写 |
| 识别引擎 | 本地 Whisper（large-v3 级别），离线 |
| 语言 | 中英混合 |
| 说话人区分 | v1 必须有；**麦克风与系统声音两路都做** |
| 音频源 | 麦克风 + 系统声音 |
| 平台 | macOS 优先；架构可跨平台，Windows/Linux 二期 |
| 隐私 | 全本地、不联网（模型下载除外） |

## 2. 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 外壳 / UI | **Tauri 2 + Web 前端**（React 或 Svelte，实现阶段定） | 跨平台、包体小、UI 自由度高 |
| 核心引擎 | **Rust** | 音频管线、调度、状态机一套代码跨平台 |
| 识别 ASR | **sherpa-onnx（Whisper large-v3）** | 离线、中英混合好；ONNX Runtime 在 Mac 走 CoreML |
| 说话人区分 | **sherpa-onnx diarization**（pyannote 分段模型 + 说话人嵌入模型） | 与 ASR 同一依赖，跨平台 |
| VAD | **Silero VAD**（sherpa-onnx 内置） | 切分语音段，驱动分段识别 |
| 音频采集 | 平台专属模块：mac = **ScreenCaptureKit**；Win = WASAPI loopback；Linux = PipeWire monitor | 系统声音捕获天生平台相关，单独隔离 |
| 存储 | 本地文件（Markdown / JSON）+ 轻量 **SQLite** 索引 | 简单、可移植、易导出 |

> 备选优化路径：若 sherpa-onnx 的 Whisper（ONNX）在 Mac 上速度不足，可将 ASR 换成 `whisper-rs`（whisper.cpp + Metal），diarization 仍用 sherpa-onnx。接口隔离使该替换局部化。

## 3. 架构与模块边界

```
┌─────────────────────────────────────────────┐
│  Tauri 前端 (Web UI)                          │
│  录制视图 / 笔记列表 / 笔记详情                │
└───────────────▲───────────────────────────────┘
        事件流 (Tauri IPC: partial / final / status)
┌───────────────┴───────────────────────────────┐
│  Rust 核心                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐ │
│  │ 音频采集  │→ │ 处理管线  │→ │ ASR + Diar   │ │
│  │ (平台模块)│  │ VAD/重采样│  │ (sherpa-onnx)│ │
│  │ mic+系统  │  │ 16kHz单声 │  │              │ │
│  └──────────┘  └──────────┘  └──────┬───────┘ │
│                                  ┌───▼───────┐ │
│                                  │ 会话/存储  │ │
│                                  │ 笔记导出   │ │
│                                  └───────────┘ │
└────────────────────────────────────────────────┘
```

### 关键隔离点
- 唯一平台相关的部分抽象为 trait **`AudioCapture`**（输出统一的 16kHz 单声道 PCM 帧 + 来源标记）。换平台只换它的实现，其余全部复用。
- ASR 与 diarization 通过 **`Recognizer`** 接口暴露，便于将来替换后端（如 whisper-rs）。

### 模块职责（单一职责、接口清晰）
- **音频采集**：打开 mic + 系统声音两路设备，输出带 `source(mic/system)` 标记的 PCM 帧；处理设备热插拔与权限。
- **处理管线**：重采样到 16kHz 单声道；按来源缓冲；VAD 切分语音段；驱动快流 / 慢流调度。
- **ASR + Diar**：对音频段做 Whisper 识别与说话人区分；输出文本、说话人、时间戳。
- **会话 / 存储**：维护当前会议状态；定稿段边到边落盘；导出 Markdown / 纯文本；维护 SQLite 索引。

## 4. 实时显示 + 说话人区分策略（双流）

### 核心矛盾
- Whisper 非流式：对完整段识别，天然有"凑一段才出字"的延迟。
- Diarization 在完整片段上最准；逐字实时判断谁在说会抖动。

### 双流方案
```
音频(mic+系统, 16kHz 单声道) → VAD 持续切分
   ├──【快流·低延迟】每 ~1–2 秒对"当前累积段"做一次 Whisper，
   │     实时刷新"正在转写…"文字（可能微调），先不带说话人标签
   └──【慢流·定稿】当 VAD 判定一段结束(静音 > ~0.7s)：
          1. 对完整段做高质量 Whisper（最终文字）
          2. 对完整段做说话人区分 → 标上说话人
          3. 用定稿文本+说话人替换快流的临时文字，并落盘
```
**效果**：文字近实时滚动，说话人标签在每句话说完后 1–2 秒内补上并固定。

### 说话人区分细节
- **麦克风与系统声音两路都跑 diarization**（现场可能多人共用一台机，线上还有远端）。
- 两路的**声纹嵌入汇入同一次聚类**，得到**全局统一的「说话人 1…N」**；来源 mic/system 作为元数据保留。
- 人数未知 → **阈值自动聚类**，不固定人数。
- 支持给说话人**改名**，改名实时回填全文。

### Tauri IPC 事件契约
- `partial` → `{ text }`：快流临时文本，频繁刷新。
- `final` → `{ text, source: mic|system, speaker_id, start_ts, end_ts }`：定稿段，入库。
- `status` → `{ state: recording|paused|loading_model|error, device, lag_seconds? }`。

## 5. 界面

1. **录制视图（核心）**
   - 顶部：开始 / 暂停 / 停止、计时、音频源开关（麦克风 / 系统声音）、电平表。
   - 主体：实时转写流，按说话人分段、带时间戳；临时文字浅色，定稿后变实色并补说话人。
   - 侧栏：说话人列表，可点击改名（实时回填全文）。
2. **笔记列表**：历史会议（标题 / 日期 / 时长 / 说话人数），支持全文搜索。
3. **笔记详情**：完整带说话人时间轴的转写，可编辑、导出。

## 6. 笔记存储

- 每场会议一个文件夹：
  - `transcript.json`：结构化（段落 / 说话人 / 时间戳 / 来源）。
  - 导出的 `.md` / `.txt`。
- 轻量 **SQLite** 仅做索引（标题、日期、全文搜索），不存大数据。
- 默认路径 `~/Library/Application Support/<app>/notes/`，设置中可改。
- **导出**：v1 做 **Markdown + 纯文本**；SRT/VTT 字幕二期。

## 7. 模型管理

- 首次启动**引导下载模型**（Whisper large-v3 + Silero VAD + 说话人分段/嵌入模型），显示进度、断点续传、完整性校验。
- 模型存 `Application Support`；large-v3 约 1GB+，设置中可切换 medium 做"快但略糙"模式。
- **不内置进安装包**，保持 App 体积小、规避分发体积问题。

## 8. 错误处理（最高原则：会议进行中不丢内容）

| 场景 | 处理 |
|---|---|
| 未授予麦克风 / 屏幕录制权限 | 启动检测，引导到系统设置授权；缺系统声音权限时降级为"仅麦克风"继续 |
| 模型未下载 / 下载中断 | 阻塞进入录制；断点续传，校验失败可重试 |
| 录制中崩溃 / 断电 | **边录边落盘**：定稿段实时写入 `transcript.json`，重启后恢复未导出的会议 |
| 识别 / 区分某段失败 | 该段标 `[识别失败]` 占位，不影响后续；可对该段重试 |
| 音频设备切换（插拔耳机） | 监听设备变化，自动重连，UI 提示 |
| 性能跟不上（实时落后） | 快流自动降频 / 降模型，保证慢流定稿不丢；UI 显示"处理中 N 秒待补" |

## 9. 测试策略

- **Rust 核心单元测试**：VAD 切分、重采样、双流缓冲、说话人聚类标签稳定性 —— 用**预录音频样本**（中英混合、多说话人、静音段）做确定性测试，不依赖真实麦克风。
- **ASR / Diar 集成测试**：固定样本音频 → 断言段落数、说话人数、关键词命中（容忍小幅文本差异）。
- **音频采集**：mock `AudioCapture` trait 喂样本；真实设备做手动冒烟测试。
- **前端**：转写事件流渲染、说话人改名回填、导出格式快照测试。
- **端到端冒烟**：脚本播放样本会议音频经虚拟设备 → 跑完整管线 → 校验导出 Markdown。

## 11. 开放问题 / 实现期再定

- 前端框架最终选型（React vs Svelte）。

- 说话人改名后是否持久化映射到 `transcript.json`（倾向：是）。
