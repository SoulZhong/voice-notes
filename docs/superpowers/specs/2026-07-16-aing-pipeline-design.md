# Aing — AI 增强流水线与知识图谱 架构设计

日期:2026-07-16
状态:架构设计(整体定架构,分七阶段各自 plan→实现)

## 背景与命名

现有 LLM 阶段(旧称「精修」)只在**停录后对整篇**做文本清理(错字/口头语/实体归一/排版),per-note、一次性、录制结束才跑。

产品方向升级为一条 AI 增强流水线,统一命名为 **Aing**(专有词,弃用「精修」;动词化:「Aing 中 / 已 Aing」)。Aing 做四件相互关联的事:

1. **修订**:ASR 每产出几大段就交大模型做**语义连贯性修订**(校正错字/口头语/指代,确保读得通、意思对)——从「停录后整篇」改为「录制中分批、实时可见」。
2. **抽取关键实体**:人/组织/项目/术语/决议/任务…
3. **更新知识图谱**:实体入库(全局),笔记↔实体、实体↔实体建边。
4. **建立笔记间关联**:据共享实体把笔记连起来。

四块被当作**一套连贯架构**设计(只抠一小块会考虑不周),实现分阶段交付。

## 目标 / 非目标

**目标**:把 app 升级为「带实时 AI 处理的个人知识库」——录制中边录边 Aing(修订文本实时替换、实体实时浮现),沉淀一个跨所有笔记的知识图谱,并以此关联笔记。

**非目标**:重做 ASR / 声纹 / 说话人聚类本身(复用现有);离线/端侧大模型(Aing 引擎走云端 HTTP);多用户/云同步(仍纯本机)。

## 核心原则:大模型是纯增量层,缺失不塌

**未配置任何大模型服务时,app 的核心工作照常完整完成。** 录音、ASR 转写、说话人区分、本地成稿(过滤 + 全局重聚类 + 分段)这条链**不依赖大模型**,任何时候都产出一份可用的笔记稿。Aing(修订 / 抽实体 / 图谱 / 关联)严格叠在其上:

- **无大模型** → Aing 不运行:笔记正文 = 本地转写稿(不修订)、无实体/图谱/相关笔记。核心链路、会议搭子、录音回放等一切照常。
- **有大模型** → Aing 在本地稿之上增量修订并沉淀图谱。
- 绝不能出现「因为没配大模型,笔记做不出来/录音用不了」。这是每个阶段的验收红线:Aing 相关代码路径的任何失败(未配置/超时/连不上/结构化解析失败)都只降级、不影响本地成稿(延续现有「精修失败保留原文」的姿态)。

## 关键决策(已拍板)

- **引擎双模**:**HTTP 大模型**(快、结构化输出)做**实时分批** Aing;选 **Agent CLI** 则退化为**停录后整篇**(无实时、无增量实体),兼容旧配置。Agent CLI 整篇十几分钟,做不了实时。
- **实时可见**:录制中笔记正文「原始→修订版」就地替换,实体实时浮现。
- **知识图谱 = 全局个人知识库**;**「人」类实体复用现有 person_id/声纹体系**(不另建人物),仅非人实体新增。
- **全局图谱索引用 SQLite**(rusqlite,新依赖);每笔记 `aing.json` 是人读真值,SQLite 是可整库重建的派生索引。
- **建在现有生命周期 actor 之上**(单写者 + 精修维度 + 钩子总线的自然延伸),不另起并行管线抢写笔记。

## 架构

### 1. 实时管线(数据流)

```
录音中:麦克风 → ASR → 段落逐条落定
                        │  每积累 N 段(按说话人轮次/停顿边界凑批,避免切断半句)触发一批
                        ▼
   Aing 批处理(HTTP 大模型,一次调用同时产出结构化 JSON):
     输入 = 本批原始段 + 携带上下文(术语表 + 已识实体清单 + 上一批修订尾巴)
     输出 = revised[](每段修订文本) + entities[](类型/名称/别名/提及段落下标) + relations[](可选,后期)
                        │                              │
              经 lifecycle actor 单写者回写         并入全局知识图谱(单独事务)
              替换该批段落文本(实时可见)           人→复用 person_id;新非人实体入库
                        ▼                              ▼
                录制页文本"原始→修订"就地替换       相关笔记随共享实体实时更新
```

设计点:
- **触发单元**:积累 N 段已落定段落,按说话人轮次/停顿边界凑批;批幂等(重跑同结果)。
- **跨批连贯**:每批把「术语表 + running 实体清单 + 上一批修订尾部」作为上下文前传(扩展现有 `llm.rs` 的术语表前传),保证指代/术语跨批一致。
- **单写者回写**:修订文本与实体只经 **lifecycle actor** 回写(它是唯一 note writer,天然无并发写冲突);ASR 产新段与 Aing 回写旧段并行不打架。
- **停录收尾**:仍做**全局重聚类**(说话人跨全篇校正,本就必须整篇)+ 一次**全篇连贯性终审**(缝合分批修订)。
- **Agent 模式**:退化为停录后整篇(旧行为),无实时、无增量实体。

### 2. 数据模型

**实体记录**
```
Entity {
  id            // 人实体 = 复用 person_id;非人实体 = 新分配 ent_id
  kind          // person(复用) | org | project | term | decision | task | place | date...
  name          // 规范名(canonical)
  aliases[]     // 别名/同音异写(归一并入)
  summary       // 大模型维护的一句话简介(实体页用)
  created_at / updated_at
}
```

**存储:文件为真值、图谱为派生索引**(沿用 app「盘上为真、可重建」惯例,如声纹库)
- **每笔记 `aing.json`**(取代 `refined.json`):修订后段落 + 每段实体提及区间(正文高亮用)+ 本篇实体清单。人读友好、单笔记自洽。
- **全局图谱索引(SQLite)**:实体注册表 + `笔记↔实体` 边(含提及位置)+ `实体↔实体` 关系。录制中增量写、支撑实体页/相关笔记/搜索/图谱查询;**从所有 `aing.json` 派生、可整库重建**(丢了不丢数据)。

**实体解析(「更新知识图谱」的核心)**:大模型吐的是**原始提及**;解析层落到规范实体:① 人→匹配 person 系统;② 非人→按 name/alias 在注册表找;③ 找不到→建新;④ 别名/指代归一、疑似重复合并(与「会议搭子」同名合并同源)。保证图谱不长满重复节点。文本人物先建、日后声纹对上可合并。

**术语与迁移**:`refined.json`→`aing.json`;`stages.llm`→`stages.aing`;全线「精修」文案/命名→ **Aing**;旧 `refined.json` 一次性迁移。

### 3. UI 表面(修订后 IA)

左侧页签:**录音 / 会议搭子 / 图谱(新) / 钩子 / AI / 设置**。

- **录制与详情合一的笔记页**(录制页并入笔记详情页):录制中实时更新、结束即详情。含:正文**实体高亮**(可点,人→会议搭子、非人→实体页)、底部**相关笔记**(共享实体派生,标出共享了谁)、**Aing 状态**(替换原精修 banner:`Aing 中… / 已 Aing / 部分失败`)、录制中**实时实体 chips**。
- **图谱页(新,实体页+图谱合一)**:实体浏览 + 搜索/过滤(按类型分组)+ **力导向图可视化**(节点=实体/笔记,边=关系/共现,拖拽探索);非人实体详情从这里进。
- **会议搭子(保留)**:人物系统 UI 不动;人实体即会议搭子详情,增补「相关实体 / 相关笔记」。
- **AI 设置**:「智能精修」区改为 Aing 引擎配置(HTTP/Agent 双模、实时开关、频率/预算)。
- **无大模型时的空状态**:笔记页不显示 Aing 状态/实体高亮/相关笔记(正文=本地转写稿);图谱页显示引导空态(「配置大模型以启用知识图谱」+ 跳设置);会议搭子照常。所有 Aing 表面**优雅隐藏**而非报错。

### 引擎调用的代价(需持续留意)
录制中持续调云端大模型 = ① 每批**付费调用**(长会议成本上升);② 转写文本**实时持续离开本机**上云(隐私)。P5 实时阶段需提供频率/预算控制(如每 N 分钟或累计 M 字才抄一次)。

## 分期实施路线(合并为 3 阶段,避免返工/冗余)

**去冗余原则**:Aing **批函数**(一批文本 + 上下文 → 结构化「修订+实体」)是可复用核心;停录后编排与实时编排**都调它**。停录后路径是**永久**的(Agent 恒走它、HTTP 非实时也走它),非丢弃品。数据模型(`aing.json` 含提及区间、SQLite 图谱 schema)**一次定型**,UI 与实时都建其上,不返工。

| 阶段 | 内容 | 交付价值 | 依赖 |
|---|---|---|---|
| **Phase 1 — Aing 后端 + 知识图谱** | 改名 精修→Aing(`refined.json`→`aing.json` + 迁移、`stages.aing`);**可复用批函数**(HTTP 结构化:修订 + 实体;缺模型/失败降级);`aing.json` 一次定型(修订段落 + 提及区间 + 本篇实体);**SQLite 全局图谱** + 实体解析(复用 person、去重合并),从 `aing.json` 派生可整库重建;**编排先走停录后整篇**(HTTP/Agent 同走,永久路径)。 | Aing 修订 + 实体 + 图谱落库 + 笔记↔实体建边(停录后) | — |
| **Phase 2 — UI** | 合一笔记页(实体高亮 + 相关笔记 + Aing 状态)+ 图谱页(实体浏览/搜索/过滤 + 力导向可视化)+ 会议搭子增补相关实体/笔记;缺模型时全表面优雅隐藏/引导空态。 | 图谱可见:高亮、相关笔记、跨笔记浏览与图谱 | Phase 1 |
| **Phase 3 — 实时增量** | 在 Phase 1 批函数之上加:录制中分批触发(说话人轮次凑批)、单写者经 actor 实时回写(原始→修订就地替换)、实时实体 chips、频率/预算 + 隐私开关。纯附加,复用批函数与数据模型。 | 「边录边 Aing」实时体验 | Phase 1,2 |

取舍:最强调的「实时」落 Phase 3——须站在 Phase 1 批函数 + 图谱之上才稳;Phase 1–2 已交付「Aing 修订 + 实体 + 图谱 + 相关笔记 + 可视化」(停录后、非实时),各自可发版。Phase 1 的停录后编排在 Phase 3 被复用而非替换,故无返工。

## 风险 / 待解

- **跨批连贯**:分批修订可能损全局连贯;靠上下文前传 + 停录终审兜,效果需实测。
- **实体解析准确度**:错并/漏并影响图谱质量;需人可干预的合并/拆分(复用会议搭子合并交互)。
- **并发回写**:ASR 产新段 vs Aing 回写旧段,须严格走 actor 单写者;段落下标在 recluster 前后的稳定性要设计。
- **成本/隐私**:实时云调用,P5 必带频率/预算与隐私开关。
- **迁移**:存量 `refined.json` → `aing.json` 的一次性迁移与回滚。
- **SQLite 依赖**:引入 rusqlite;图谱可整库从 `aing.json` 重建以防损坏。

## 下一步
每阶段各自走 plan→实现,从 **Phase 1(Aing 后端 + 知识图谱)** 起。Phase 1 若单个 plan 过大,可按「改名 / 批函数+引擎 / 图谱+解析」拆成前后衔接的 plan,但仍属同一阶段、共用一次定型的数据模型。
