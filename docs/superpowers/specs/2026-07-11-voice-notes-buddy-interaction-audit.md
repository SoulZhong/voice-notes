## 一、现状地图(谁在读写会议搭子)

### 数据层 `store/voiceprints.rs`
- Person = { name(空=未命名), 每信道质心, total_ms, last_seen } + redirects(合并重定向) + 录音样本(≤5 份 WAV)。
- 写入口:开录种子注入 → 录制中实时入库(无主簇 ≥10s 领 P 号) → 停止 upsert 质心回写/建档+补样本;rename / merge(质心加权并入+样本迁移+链条压扁) / delete(连删样本);数据层**不禁止重名**。
- 读出口:list_people、笔记 load 只读 join(S* 本地名优先)、get_refined 只读 join(跟随库现名)、精修种子命名。

### 界面触点
| 触点 | 能做什么 | 落点 |
|---|---|---|
| 侧栏「会议搭子」页签 | 人物索引(待命名/已命名分组、色点、最近出现) | 只读 |
| /speakers 概览 | 统计(总数/已命名/待命名)+引导文案 | 只读 |
| /speakers/[id] 详情 | 改名、原声试听(多样本)、合并到…(二段确认)、删除(二段确认);录制中禁合并/删除 | rename_person / merge_person / delete_person |
| 笔记页·精修稿 chips(新) | 编辑面板:改名(同步库现名)、这是我、从搭子选人关联 | rename_refined_speaker / assign_refined_person |
| 笔记页·原始稿 chips | 编辑面板:改名(**仅本笔记**)、这是我;段落徽章可换说话人 | rename_speaker(本地) |
| 录制页 live chips | 同原始稿(writer 路径);种子命中自动显名 | rename_speaker(writer) |
| 导出 MD/TXT | **恒导原始稿**(segments+speakers join 名) | export_note |
| MCP/CLI | list_speakers(带 note_count)、get_note(prefer_refined);**无任何写工具** | 只读 |

## 二、缺口与增强规划(按优先级)

### P1-1 改名重名处置(本次点名的例子)
现状:所有改名入口都不查重名。库里出现两个「张三」后,选人面板/侧栏/种子命名全部
无法区分;而**重名十有八九=同一个人被声纹拆重**,正确动作是合并而不是留两个同名。

方案(数据层继续允许重名,全部在交互层处置;比较用 trim 归一):
- **详情页改名撞名**:不直接提交,弹行内确认(menu 语言,非系统对话框——改名非破坏):
  「已有一位『张三』(最近出现 X · 累计 Y)。是同一个人吗?」
  [是,合并成一个](当前人并入已有者,跳转已有者详情) / [不是,保留同名] / [取消]。
- **chips 面板改名撞名**(精修稿/原始稿同一套):
  - 该说话人未关联 person → 面板内联提示条「会议搭子里已有『张三』」,主按钮改为
    **[关联他]**(=assign,等价于"这个说话人就是库里的张三"),次按钮 [只改名字]。
  - 已关联 personA 而撞 personB 名 → 提示「另一位搭子也叫张三,可能是重复条目」,
    给 [去详情页合并] 链接 + [仅改名];面板内不做合并(破坏性动作不进菜单,DESIGN 规则)。
- **展示兜底**(即使用户选择保留重名):凡显示人名的列表(侧栏/选人面板/合并菜单),
  同名条目行尾追加区分后缀「· 最近 MM-DD」。
- MCP/CLI 将来有写工具时不做确认(工具语义直改),文档说明即可。

### P1-2 导出跟随精修稿
现状:用户看着精修稿点「导出 MD/TXT」,得到的是原始逐字稿——所见非所得。
MCP `get_note` 已有 `render_refined` 渲染逻辑。
方案:export_note 增 `prefer_refined`(默认跟随当前视图),渲染复用 render_refined;
文件名区分(transcript.md 精修 / transcript-raw.md)或原名覆盖+页面提示,拍板其一。

### P1-3 「这是我」重名收口
「这是我」=改名"我";两个不同 person 先后被标"这是我"→ 库里两个「我」。
无需引入"本人"概念,P1-1 的重名机制天然覆盖(第二次标记时提示合并)。

### P2-1 原始稿/录制页也能"选人关联"
现状:只有精修稿能从搭子选人;原始稿 S* 只能本地改名,不一致。
方案:新命令 `assign_note_speaker_person(note_id, speaker_id, person_id)`——写
speakers.json 的 person_id 并清空本地 name(join 显库名);活动会话走 writer 锁路径
(与 rename_speaker 同模式),录制页面板也传 people。收益:录完/录制中即可认人,
不必等精修。注意:改 person 关联会影响停止时 upsert 回写方向,需在 registry 侧同步
簇的 person 绑定(录制中场景),首版可先只支持非活动笔记,录制中禁用选人区。

### P2-2 选人面板与列表排序
- listPeople 返回 BTreeMap 序(P1,P10,P2… 字典序),侧栏与选人面板顺序反直觉。
  统一:已命名在前按 last_seen 降序,未命名殿后;后端排或前端排拍板(建议后端,三处同源)。
- 选人面板未命名条目只有「说话人 N」+色点,难辨认 → 行尾补「最近 MM-DD」次要文案。

### P2-3 详情页「出现过的会议」
MCP list_speakers 已会扫 speakers.json 统计 note_count,GUI 详情页没有。
方案:详情页加一张卡列该人出现过的会议(标题+日期,点击跳转笔记),这是试听之外
确认"认对了人"的第二手段;后端新查询命令(person_id → 笔记摘要列表)。

### P2-4 合并体验
- 合并确认里预览结果:「合并后名字为 X,累计发声 Y」——现在名字继承规则
  (winner 无名才继承 loser 名)对用户是黑盒。
- 方向建议:默认把数据薄的一方并入厚的一方(total_ms 大者为 winner),UI 上仍可换向。
- 概览页「疑似重复」提示卡:检测同名 person,给合并入口(与 P1-1 呼应,处理存量重名)。

### P2-5 重新精修 vs 手工改名
「重新精修」整写 refined.json:已关联 person 的名字经 join 保留,**未关联的手工改名会被冲掉**。
方案(递进):①重新精修按钮旁提示「未关联搭子的手工改名会丢失」;②改名时若未关联,
面板轻推关联(P1-1 的 [关联他] 已顺带解决大半);③远期:refined 局部保留用户改名(schema 加 user_named 标记)。

### P3(记账,不阻塞)
- 删除人物的确认文案补后果说明:「历史笔记中该人恢复显示为编号」。
- 样本按钮加录制日期标注;老人无样本时提供"从某场会议截一段作样本"(复杂,backlog)。
- MCP/CLI 写工具对等(rename/assign/merge)——agent-native parity,需先拍板门控语义
  (内容编辑是否归入 mcp_allow_control)。
- 录制中实时选人(P2-1 的活动会话版,涉及 registry person 重绑定,单独设计)。

## 三、建议排期

1. **第一批(改名闭环)**:P1-1 重名处置 + 同名展示后缀 + P1-3;顺带 P2-2 排序(小)。
2. **第二批(所见即所得)**:P1-2 导出跟随精修稿。
3. **第三批(关联补全)**:P2-1(先非活动笔记)+ P2-3 出现过的会议。
4. **第四批(治理)**:P2-4 合并体验 + P2-5;P3 逐项收。

不在本次范围:聚类精度(A2,挂起待 golden 真值数据,见项目记忆);说话人分离算法本身。
